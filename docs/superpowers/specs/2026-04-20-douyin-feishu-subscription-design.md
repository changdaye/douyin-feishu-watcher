# 抖音博主更新订阅到飞书机器人 - 设计文档

## 1. 项目目标

构建一个长期运行的小型服务，持续监控少量公开抖音博主（预计 7~8 个，不超过 20 个）是否发布了新视频；一旦发现新视频，在 1 小时内将更新推送到指定的飞书机器人。

## 2. 已确认范围

### 2.1 需要实现
- 监控少量公开抖音博主的新视频更新
- 在 1 小时内将新视频通知推送到飞书机器人
- 部署在现有火山云服务器（2 核 2G）上长期运行
- 以稳定、少维护为第一优先级
- 使用 Python 实现

### 2.2 明确不做
- 不追求飞书内直接播放抖音视频
- 不下载视频到服务器
- 不做复杂后台管理系统
- 不做大规模账号池、代理池、反风控对抗系统
- 不做公开视频历史补发

## 3. 核心约束与设计原则

### 3.1 核心约束
- 订阅对象是公开博主，而非用户自有账号或授权账号
- 目标是长期稳定运行，而不是追求极限实时性
- 推送时效要求为小时级（1 小时内）
- 服务器资源有限，应避免重型依赖和复杂架构

### 3.2 设计原则
- 优先选择低频轮询、轻量存储、可恢复任务模型
- 优先减少外部依赖数量和运行复杂度
- 优先保证去重准确，避免重复推送
- 优先保证故障可观察，出现异常时能够及时发现

## 4. 总体架构

系统拆分为 4 个小模块：

1. **博主管理模块**
   - 保存需要监控的博主配置
   - 管理启用/停用状态

2. **更新检测模块**
   - 周期性抓取每个博主的最近视频信息
   - 解析出视频标识、标题、发布时间、链接等字段

3. **去重与状态记录模块**
   - 持久化记录已发现视频
   - 判断某条视频是否已经推送过

4. **飞书通知模块**
   - 将新视频组装为飞书机器人消息
   - 支持正常消息发送与失败降级处理

整体运行方式为单机常驻服务，由定时任务周期触发巡检流程。

## 5. 更新检测策略

### 5.1 策略选择
采用“**低频轮询 + 页面解析 + 本地去重**”的方式。

### 5.2 轮询频率
- 默认每 30 分钟巡检一次
- 如后续验证稳定且需要更快时效，可调整为每 15 分钟一次

该频率可以满足 1 小时内推送的目标，同时降低访问频率，尽量减少页面结构变化或访问限制带来的维护压力。

### 5.3 检测流程
每轮任务执行流程如下：

1. 读取启用状态的博主列表
2. 对每个博主依次抓取其最近的视频信息
3. 解析出视频唯一标识、标题、发布时间、链接
4. 与本地已知视频记录比对
5. 将未出现过的视频判定为新视频
6. 调用飞书通知模块推送
7. 更新本地数据库中的发现状态和推送状态

### 5.4 首次启动策略
首次部署时：
- 抓取当前最近视频作为基线
- 仅建立已知视频记录
- 不补发历史视频

这样可以避免首次上线时向飞书发送大量旧内容。

### 5.5 容错策略
- 单个博主抓取失败，不影响其他博主继续执行
- 失败记录日志，等待下一轮重试
- 对连续失败的博主触发异常告警

## 6. 数据存储设计

### 6.1 存储选型
使用 **SQLite** 作为本地持久化数据库。

原因：
- 数据规模小
- 单机部署
- 无需额外维护数据库服务
- 备份与迁移成本低

### 6.2 数据分类
系统至少持久化以下三类数据：

#### A. 博主配置
字段建议：
- `id`
- `name`
- `profile_url`
- `creator_uid`（如后续能够稳定提取）
- `enabled`
- `created_at`
- `updated_at`

#### B. 视频发现记录
字段建议：
- `id`
- `creator_id`
- `video_id`
- `title`
- `publish_time`
- `video_url`
- `cover_url`（可选）
- `first_seen_at`
- `notified_at`
- `notify_status`

#### C. 巡检运行记录
字段建议：
- `id`
- `started_at`
- `finished_at`
- `checked_creator_count`
- `success_creator_count`
- `failed_creator_count`
- `new_video_count`
- `error_summary`

### 6.3 去重规则
主去重键采用：
- `video_id`

如遇到解析策略调整导致单一标识不稳定，可增加兜底判断：
- `creator_id + video_url`

系统对外汇报仍以 `video_id` 为主判定标准。

## 7. 配置设计

### 7.1 服务配置
通过 `.env` 管理以下配置：
- 飞书 webhook 地址
- 巡检间隔
- HTTP 请求超时时间
- 日志级别
- 异常告警阈值
- SQLite 文件路径

### 7.2 博主列表配置
第一版使用独立静态文件维护博主列表，推荐：
- `creators.yaml` 或 `creators.json`

原因：
- 数量少
- 变更频率低
- 不需要后台管理界面

后续如果要支持页面化配置，再逐步迁移到数据库管理。

## 8. 飞书消息设计

### 8.1 消息内容
每条新视频通知包含：
- 博主名
- 视频标题
- 发布时间
- 抖音链接

可选增强字段：
- 封面图

### 8.2 消息形态
推荐优先级：
1. 卡片消息
2. 富文本（post）消息
3. 纯文本消息

实际策略：
- 正常情况下发送卡片消息
- 卡片发送失败时自动降级为文本消息

### 8.3 推送粒度
每条新视频单独发送一条消息。

原因：
- 订阅量少，不会造成明显刷屏
- 单条消息更易阅读和定位
- 有利于后续排查重复推送或漏推问题

## 9. 异常处理与告警

### 9.1 抓取异常
典型场景：
- 页面不可访问
- 请求超时
- 页面结构变化
- 解析失败

处理方式：
- 记录错误日志
- 本轮跳过该博主
- 下一轮自动重试

### 9.2 推送异常
典型场景：
- 飞书 webhook 失效
- 请求超时
- 频率限制
- 消息格式错误

处理方式：
- 进行有限重试（建议 1~3 次）
- 仍失败则记录推送失败状态
- 计入异常统计并触发必要告警

### 9.3 数据异常
典型场景：
- 标题缺失
- 发布时间为空
- 链接异常

处理方式：
- 优先采用容错拼装消息
- 无法满足基本推送字段时跳过并记录错误

### 9.4 告警规则
满足任一条件时，向飞书发送系统异常通知：
- 某个博主连续抓取失败达到阈值（建议 3 次）
- 全部博主在同一轮全部失败
- 飞书 webhook 连续发送失败
- SQLite 写入异常

## 10. 技术选型

推荐技术栈：
- **Python 3.11+**
- **APScheduler**：定时任务调度
- **httpx**：HTTP 请求
- **BeautifulSoup / lxml**：HTML 解析
- **SQLite**：轻量持久化
- **logging 或 loguru**：日志
- **systemd**：Linux 常驻运行管理

选型原则：
- 保持依赖少、结构清晰
- 优先选用成熟稳定的 Python 工具
- 优先适配单机轻量部署模型

## 11. 代码结构建议

建议第一版项目结构如下：

```text
project/
  app/
    config.py
    models.py
    fetcher.py
    parser.py
    deduper.py
    notifier.py
    scheduler.py
  main.py
  creators.yaml
  .env
  data/
    app.db
  logs/
```

模块职责：
- `config.py`：读取配置
- `models.py`：数据库访问
- `fetcher.py`：抓取博主页面
- `parser.py`：解析视频字段
- `deduper.py`：去重逻辑
- `notifier.py`：飞书消息发送
- `scheduler.py`：定时任务编排
- `main.py`：应用启动入口

## 12. 部署设计

目标部署环境：火山云 2 核 2G Linux 服务器。

推荐部署方式：
- 使用 `python -m venv` 创建独立虚拟环境
- 使用 `systemd` 托管进程
- 用 `.env` 保存敏感配置
- SQLite 文件保存在本地磁盘
- 日志写入本地文件，便于回溯

## 13. 测试与验证策略

至少覆盖以下三类验证：

### 13.1 页面解析测试
验证是否能从目标页面中提取出：
- 视频唯一标识
- 视频标题
- 发布时间
- 视频链接

### 13.2 去重测试
验证同一条视频不会被重复推送。

### 13.3 飞书推送测试
验证构造出的消息可以成功发送到飞书机器人。

### 13.4 联调测试
上线前执行：
- 手动触发一轮巡检
- 核对新视频识别是否正确
- 核对飞书消息格式是否符合预期
- 模拟飞书 webhook 异常和页面抓取异常

## 14. 后续实现边界

第一版实现重点是“稳定可用的订阅通知器”，而不是通用内容平台。

只有在以下需求明确出现时，才进入第二阶段扩展：
- Web 管理界面
- 多机器人分发
- 每日汇总报表
- 更复杂的异常恢复机制
- 多平台内容订阅

## 15. 推荐实施结论

对于当前目标，推荐实现路线为：

**Python + APScheduler + httpx + BeautifulSoup/lxml + SQLite + 飞书机器人 webhook + systemd 单机部署**

这是当前在“少量博主、小时级通知、尽量稳定、少维护”条件下，最符合成本与可靠性平衡的第一版方案。
